# Design Decisions

A running log of the non-obvious engineering choices, with the reasoning behind
each. Written for a reviewer who wants to know *why*, not just *what*.

## 1. PostgreSQL is the queue (no Redis / RabbitMQ)

The core queue is a `jobs` table claimed with `SELECT ... FOR UPDATE SKIP LOCKED`.
Reasons:

- **One source of truth.** Job state, payload, execution history, and the "ready
  to run" set all live in one transactional store. There is no dual-write problem
  between a queue broker and a metadata DB, and no reconciliation job to keep them
  consistent.
- **Exactly-the-guarantees-we-need.** `SKIP LOCKED` gives us atomic, contention-free
  claiming across N workers without a broker. Delayed/scheduled jobs are just rows
  with a future `run_at` â€” no separate "delayed set" abstraction.
- **The assignment weights DB design + concurrency heavily**, so the DB-backed queue
  is the showcase rather than an implementation detail hidden inside Redis.

Trade-off: throughput ceilings are lower than a purpose-built broker, and we poll
rather than push. Documented as future work: `LISTEN/NOTIFY` to cut claim latency.

## 2. The partial claim index

```
CREATE INDEX ix_jobs_claim ON jobs (queue_id, priority DESC, run_at, id)
WHERE status = 'queued';
```

- **Partial** (`WHERE status = 'queued'`): the index only contains rows that are
  actually claimable. Completed/dead/running jobs â€” the vast majority over the
  lifetime of the system â€” never bloat it, so it stays small and hot.
- **Column order mirrors the claim `ORDER BY`** exactly: `queue_id` (equality
  filter) â†’ `priority DESC` â†’ `run_at ASC` â†’ `id ASC`. Postgres can satisfy the
  ordering straight from the index and stop at `LIMIT`, no sort node.
- `id` as the final tiebreaker gives stable FIFO ordering among jobs of equal
  priority and `run_at`.

Supporting indexes and their justification:

| Index | Purpose |
|-------|---------|
| `ix_jobs_queue_status` | per-queue status counts for `/stats` |
| `ix_jobs_scheduled_run_at` (partial `WHERE status='scheduled'`) | scheduler's due-promotion scan |
| `ix_jobs_idempotency` (partial unique) | one live job per `(project, idempotency_key)` |
| `ix_jobs_project_created` | project-scoped listing with date-range filters |
| `ix_executions_started_at` | metrics bucketing over recent executions |
| `ix_scheduled_next_run` (partial `WHERE is_active`) | scheduler's due-cron scan |
| `ix_workers_heartbeat` | reaper's stale-worker scan |

## 3. Primary keys: bigint identity, not UUID

`GENERATED BY DEFAULT AS IDENTITY` bigints on every table. Chosen over UUIDs because:

- The claim index's final tiebreaker is `id ASC`, used for **FIFO fairness** among
  otherwise-equal jobs. That only works if ids grow monotonically with insertion
  order â€” random UUIDs would destroy FIFO semantics.
- Bigints are 8 bytes vs 16 for UUIDs, and monotonic inserts keep the btree's
  right edge hot, minimizing page splits on the highest-churn table (`jobs`).
- Ids are internal; we never expose them across a trust boundary where UUID
  unguessability would matter (projects carry a separate random `api_key`).

## 4. `timestamptz` everywhere

Every timestamp column is `TIMESTAMP WITH TIME ZONE`. Workers, scheduler, and API
may run in different timezones; storing UTC instants (which `timestamptz` does)
makes `run_at <= now()` comparisons unambiguous and correct regardless of the
session timezone. `now()` in the claim/promotion queries is evaluated by the DB, so
there is a single clock â€” the database's â€” for all "is it due yet" decisions.

## 5. Native Postgres enums for status columns

`job_status`, `execution_status`, `worker_status`, `retry_strategy`, and `org_role`
are native enum types, not free-text. This enforces the state machine's vocabulary
at the schema level â€” an illegal status literally cannot be stored. The baseline
migration explicitly `DROP TYPE`s them on downgrade, since `DROP TABLE` alone leaves
the types behind and would break a re-`upgrade`.

## Cascade rules

- `projects â†’ queues â†’ jobs`: `ON DELETE CASCADE`. Deleting a project tears down its
  whole subtree; there are no orphaned queues or jobs.
- `jobs.worker_id`: `ON DELETE SET NULL`. A worker row disappearing must never
  cascade-delete real job history.
- `queues.retry_policy_id`, `jobs.retry_policy_id`: `SET NULL` â€” losing a policy
  falls back to defaults, never destroys jobs.

## Environment / tooling deviations (from the documented setup)

The reference commands assume `docker compose` on PATH and a system Python 3.12.
On the build machine:

- **Python 3.12 was provisioned via `uv`** (`uv python install 3.12`) because the
  system only had 3.11 and 3.14. The `.venv` targets 3.12.12, matching the stack
  requirement. Dependencies installed with `uv pip install -r requirements.txt`.
- **Docker Desktop was installed but not on PATH**; commands prepend its
  `resources/bin` directory. The daemon and `docker compose` otherwise behave as
  documented.

These are local-machine details and do not affect the application code or the
documented `docker compose` / `alembic` / `uvicorn` workflow.

- **`bcrypt` pinned to `4.0.1`.** `passlib` 1.7.4 (latest) probes the bcrypt
  backend at startup with a >72-byte test hash; bcrypt â‰Ą4.1 raises `ValueError`
  on that probe, breaking password hashing entirely. `4.0.1` is the last release
  passlib detects cleanly. We keep the `passlib[bcrypt]` stack the assignment
  specifies rather than swapping the hashing library.

## Auth & error handling

- **JWT access tokens**, `sub = user.id` (string, as JWT requires). Registration
  bootstraps a user + a personal organization + an owner `org_members` row in one
  unit of work, so a user always has an org to own projects under.
- **Uniform error envelope** â€” `{"error": {"code", "message", "details"}}` â€” is
  produced by exception handlers for our `AppError` hierarchy, pydantic validation
  errors (`VALIDATION_ERROR`), raw `HTTPException`s, and any unhandled exception
  (`INTERNAL_ERROR`, logged with a traceback). Auth failures deliberately use
  generic codes (`INVALID_CREDENTIALS`) so they don't leak whether an email exists.
- **Structured JSON logging** with a request-id middleware: every request gets an
  id (honoring an inbound `X-Request-ID`), echoed on the response and attached to
  every log line via a `contextvar`.

## Concurrency & reliability

### `SELECT ... FOR UPDATE SKIP LOCKED` vs advisory locks

Claiming uses row-level `FOR UPDATE SKIP LOCKED` on the `jobs` rows themselves.
The alternative â€” a Postgres **advisory lock** per job id â€” would also work, but:

- `SKIP LOCKED` locks exactly the rows the claim query selects, in the same
  statement that flips their status. There's no second round-trip and no separate
  lock namespace to reconcile with row state.
- Advisory locks are session-scoped and easy to leak (a crashed worker holding an
  advisory lock needs explicit cleanup); row locks are released automatically when
  the claim transaction ends.
- `SKIP LOCKED` composes naturally with `ORDER BY ... LIMIT`, giving priority-ordered
  batch claiming for free off the partial index.

So the row-lock approach is both simpler and more correct here. The claim
transaction is deliberately tiny (claim â†’ commit) so locks are held for microseconds.

### Per-queue concurrency is a *soft* limit

At claim time a queue is excluded when
`count(jobs WHERE queue_id = q.id AND status IN ('claimed','running')) >= concurrency_limit`.
Under `READ COMMITTED`, two workers can both read a count just under the limit and
both claim, briefly overshooting. We **accept** this race rather than serialize
claims behind a heavier lock, because:

- Enforcing a hard cap would require locking the queue (or an advisory lock per
  queue), which reintroduces the contention `SKIP LOCKED` was chosen to avoid.
- The overshoot is bounded and transient â€” the next claim cycle sees the true count
  and backs off. For a throughput limiter, "approximately N in flight" is the
  standard, honest trade-off.

This is documented behavior, not a bug: the limit is a throttle, not a hard barrier.

### At-least-once semantics + idempotency

A worker can die after executing a job but before committing its `completed`
transition; the reaper will then requeue it, and it runs again. So execution is
**at-least-once**, never at-most-once. Two mechanisms make the observable result
effectively-once:

- **Client idempotency keys** â€” an optional `idempotency_key` with a partial unique
  index `(project_id, idempotency_key) WHERE idempotency_key IS NOT NULL`. Duplicate
  submits return the existing job (200, not 201). A parallel-submit race is resolved
  by the unique index: the losing insert gets an `IntegrityError`, rolls back, and
  returns the winner's job (tested under 10-way concurrency).
- **Idempotent handlers** â€” handlers receive `(job_id, execution_id, payload)` and
  are expected to be safe to re-run. The demo handlers are.

Attempt accounting keeps these honest: the **execution attempt number** is monotonic
(count of prior executions + 1), decoupled from `jobs.attempts` (the retry budget,
which resets on a DLQ "fresh" retry). This keeps the `(job_id, attempt)` unique
constraint intact even after a reset, and preserves full execution history.

### Retries are scheduled, not slept

A failed job is set back to `queued` with `run_at = now + backoff`; the worker never
`sleep()`s while holding a job. This keeps worker slots free for other work and makes
backoff survive worker restarts (the delay lives in the row, not in memory).

### Polling vs `LISTEN/NOTIFY` (future work)

Workers and the scheduler **poll** on a short interval. This is simple, robust, and
has no missed-wakeup edge cases, at the cost of a small claim latency (â‰¤ poll
interval) and steady baseline query load. A natural evolution is Postgres
`LISTEN/NOTIFY`: the API/scheduler `NOTIFY` a channel on enqueue, and idle workers
`LISTEN` to claim immediately, falling back to polling as a safety net. It would cut
latency and idle load but adds connection management and a fallback path â€” deferred
as it isn't needed to meet the reliability goals.
